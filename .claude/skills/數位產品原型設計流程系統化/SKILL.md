# 數位產品原型設計流程系統化 Skill

## 描述

此 Skill 用於將一段數位產品發想、UX 研究、提示詞迭代、User Flow、IA、Wireframe 與 Design System 的對話紀錄，萃取成一套可重複使用的產品原型設計流程。

它適合處理早期產品規劃、AI 協作產品設計、課程教學、MVP 企劃、UX 文件整理與設計系統建立等任務。

此 Skill 的最終產出是一份可直接使用的產品原型設計 Skill 文件，能幫助未來面對不同數位產品題目時，依照固定流程產出高品質設計文件與原型規格。

---

## 使用時機

適合在以下情境使用：

- 使用者想把一段產品設計對話整理成可複用方法。
- 使用者想把提示詞迭代過程變成固定模板。
- 使用者想建立可直接使用的 `SKILL.md`。
- 使用者想沉澱數位商品原型設計流程。
- 使用者想整理 UX 研究、Persona、User Journey Map、IA、User Flow、Wireframe、Design System 的標準產出流程。
- 使用者想將某個產品案例抽象化成未來可套用的工作方法。
- 使用者想避免每次重新撰寫完整產品設計提示詞。
- 使用者想把 AI 協作產品設計流程轉成可教學、可交付、可複製的 SOP。
- 使用者想從一段長對話中萃取真正有價值的任務模式，而不是只做摘要。

## 必要輸入

| 輸入項目 | 是否必要 | 說明 | 範例 |
|---|---|---|---|
| 原始對話或任務紀錄 | 必要 | 提供完整或主要的對話內容，用於萃取可複用流程 | 一段從市場痛點分析到 Wireframe 生成的完整對話 |
| 想萃取的 Skill 類型 | 必要 | 指定要整理成哪一類 Skill | 數位產品原型設計 Skill、提示詞工程 Skill、簡報生成 Skill |
| 預期使用者 | 建議 | 說明未來誰會使用此 Skill | UX 設計師、產品經理、講師、創業團隊 |
| 未來使用場景 | 建議 | 說明 Skill 未來會用在哪些任務 | 產品發想、課程教學、提案前期規劃、MVP 設計 |
| 偏好的輸出格式 | 建議 | 指定是否要使用 Markdown、表格、Mermaid、ASCII Wireframe 等格式 | Markdown 文件、SKILL.md、Mermaid 流程圖 |
| 是否需要產出 SKILL.md | 必要 | 若要建立可直接使用的 Skill，需明確要求產出 SKILL.md | 是，請產出完整 SKILL.md |
| 特定限制或風格要求 | 建議 | 指定語言、語氣、格式限制、不可使用的描述方式 | 繁體中文、專業但清楚、避免空泛描述 |
| 是否保留專案案例 | 選填 | 若需要，可將原始案例保留為示範；若不需要，應抽象化處理 | 保留家庭用藥產品作為範例，但不作為唯一用途 |
| 產出深度 | 選填 | 指定輸出要簡版、標準版或完整顧問版 | 完整顧問版 |

## 執行流程

### 1. 解析原始對話中的主要任務

讀取使用者提供的對話或任務紀錄，找出其中真正需要被沉澱的工作類型。

需辨識是否包含以下任務：

- 市場痛點分析
- Persona 建立
- User Journey Map
- MVP 產品企劃
- User Flow
- IA 資訊架構
- Lo-Fi Wireframe
- Product Design System
- Mermaid 流程圖
- 原型圖片生成
- 提示詞改寫與標準化

不要只看表面文字，需判斷背後可重複使用的產品設計流程。

---

### 2. 找出可重複使用的任務模式

從原始對話中萃取可套用到其他專案的固定流程。

若是數位產品原型設計，建議流程為：

1. 定義產品背景與目標受眾
2. 分析核心痛點與市場機會
3. 建立 Persona
4. 推導 User Journey Map
5. 進行 MVP 產品企劃
6. 建立 User Flow
7. 建立 IA 資訊架構
8. 產出 Lo-Fi Wireframe
9. 建立 Product Design System
10. 產出可視化流程圖或原型圖
11. 建立品質檢查規則

---

### 3. 移除一次性內容與不必要細節

將只適用於單一專案的資訊抽象化，避免 Skill 被限制在單一案例。

應移除或抽象化：

- 單一產品名稱
- 單一 Persona 姓名
- 單一圖片檔名
- 單次工具回應
- 暫時性的修正要求
- 過度細節化的單一流程節點

應保留：

- 任務結構
- 輸入欄位
- 輸出格式
- 品質檢查標準
- 設計限制
- 格式規範
- 可複用的判斷邏輯

若某些專案細節具有示範價值，可標註為「範例」，不可讓它成為唯一適用情境。

---

### 4. 萃取角色設定

從原始對話中抽出可重複使用的 Role 設定。

例如：

- 資深 UX 研究員
- 資深產品經理
- 資訊架構專家
- UI/UX 設計師
- Product Design System 顧問
- AI Skill 架構師
- 提示工程師
- 知識流程整理顧問

角色設定需服務任務，不可只是堆疊頭銜。

---

### 5. 萃取必要輸入條件

整理未來執行此 Skill 時，使用者應提供哪些資料。

針對數位產品原型設計，建議必要輸入包含：

- 產品名稱
- 產品類型
- 目標使用者
- 核心痛點
- 使用情境
- MVP 功能
- 商業目標
- 權限角色
- 主要任務
- 偏好的輸出格式
- 設計限制

若資料不足，應合理推論並標註「推估」。

---

### 6. 萃取標準輸出格式

將對話中反覆使用且有效的產出格式整理成模板。

可支援的標準輸出包含：

- 痛點評估矩陣
- Persona 表格
- User Journey Map 表格
- MVP 企劃表
- Mermaid User Flow
- IA Tree Diagram
- Page & Goal Matrix
- ASCII Lo-Fi Wireframe
- Product Design System 清單
- Prototype 測試建議表
- SKILL.md 文件

每種格式都需明確規定欄位、順序與限制。

---

### 7. 建立品質檢查規則

根據原始對話中的使用者偏好與任務需求，建立可重複使用的品質檢查清單。

品質標準至少包含：

- 使用繁體中文
- 結構清楚
- 不空泛
- 不只摘要
- 可直接交付
- 可被設計師、PM、工程師使用
- Mermaid 語法可渲染
- Wireframe 能支撐任務完成
- Design System 能保持頁面一致
- 若涉及健康、金融、付款、個資，需加入信任與權限提示

---

### 8. 組合成可重複使用的 Skill 文件

將萃取結果整理為獨立的 `SKILL.md`。

文件需包含：

- Skill 名稱
- 描述
- 使用時機
- 必要輸入
- 執行流程
- 輸出格式
- 品質檢查
- 限制與注意事項

文件必須脫離原始對話後仍可理解與使用。

---

## 輸出格式

每次執行此 Skill 時，標準輸出應包含以下 8 個部分：

### 1. Skill 名稱

提供一個清楚、可辨識、可重複使用的 Skill 名稱。

命名不可過於籠統，例如不要只寫「提示詞整理」。

---

### 2. Skill 描述

用 2 到 4 句說明：

- 此 Skill 解決什麼問題
- 適合處理什麼輸入
- 最終會產出什麼
- 對未來重複使用有什麼價值

---

### 3. 使用時機

用條列列出此 Skill 適合被使用的情境。

需避免列出與 Skill 無關的泛用場景。

---

### 4. 必要輸入

使用 Markdown 表格呈現。

格式如下：

| 輸入項目 | 是否必要 | 說明 | 範例 |
|---|---|---|---|

---

### 5. 執行流程

使用編號清單呈現。

每一步都需說明具體要做什麼，不可只寫抽象詞，例如「分析」、「整理」、「優化」。

---

### 6. 輸出格式

定義此 Skill 每次輸出應包含哪些部分。

可使用表格或條列。

---

### 7. 品質檢查規則

使用 checklist 呈現。

每一項需是可檢查的標準。

---

### 8. 可直接使用的 SKILL.md

最後輸出完整 Markdown 文件，並使用 Markdown code block 包裹。

---

## 限制與注意事項

- 不可逐字複製原始對話。
- 不可只做摘要。
- 不可保留過多一次性細節。
- 不可將單一產品案例誤當成唯一適用情境。
- 不可使用空泛詞彙，例如「提升體驗」、「優化流程」、「增加黏著度」；若使用，必須具體拆解成流程、欄位、頁面、元件或檢查標準。
- 若原始資料不足，需基於合理 UX 與產品設計經驗補齊，並標註「推估」。
- 若產品涉及健康、金融、付款、教育、企業資料或個資，需加入信任提示、權限控管、隱私說明、錯誤防呆與支援入口。
- 若產出 Mermaid，需避免容易導致渲染錯誤的特殊符號，並使用清楚的節點命名。
- 若產出 Wireframe，需以資訊層級、任務流程、CTA、狀態提示為主，不描述品牌視覺或精緻 UI 細節。
- 若產出 Product Design System，需讓 UI 設計師、PM 與工程師能依規範討論、設計與開發。
